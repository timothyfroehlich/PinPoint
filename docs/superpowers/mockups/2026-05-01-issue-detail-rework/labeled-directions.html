Direction 1 vs Direction 2 — labeled, with editability affordances

Same content, same data, two structures. Both use prefix labels (recognition over recall), chevrons on editable pills (affordance), real field-family colors (severity=amber, priority=purple, frequency=cyan, status=per-status). Mobile mockups include a sticky bottom comment composer so you can react to that pattern at the same time.

1 Compact horizontal pills with prefix labels + chevrons

Each editable pill prefixes the field name (Status:, Severity:, etc.) so the value is never ambiguous. Chevron ▾ + hover lift signal editability. Static fields (reporter, created, watchers) live in the subtitle line above. Status moves into the row, no longer hero — same vocabulary as the others.

Why Direction 2 wins for your audience

Direction 1 wins on…
  • Vertical compactness (~70px vs ~210px)
  • Aesthetic minimalism (Linear-adjacent)
  • Visual energy (color-rich pills cluster)
Direction 2 wins on…
  • Recognition: label + value, no field-name guessing
  • Touch targets: 44pt+ rows naturally
  • Wrap predictability: rows never re-flow awkwardly
  • Affordance clarity: chevron + structured row reads as form, not as tag
  • Familiarity (Jakob's Law): matches Jira / GitHub sidebar patterns most users have seen
  • Color-blind friendly: row label is the load-bearing identifier; color only reinforces

The core tradeoff: Direction 1 trades vertical space for aesthetic compactness; Direction 2 trades aesthetic compactness for clarity. For a non-tech audience that values being told what they're looking at, the clarity-for-space trade is correct.

Sticky bottom composer — call it. Both mobile mockups show the sticky composer pattern (always-visible "Add a comment…" input at the bottom of the viewport, like Slack/iMessage). My read: keep it for mobile. It eliminates the "scroll three viewports to comment" friction without harming the read-first flow. Desktop doesn't need it — comment input naturally sits below activity. Question: sticky on mobile — yes, no, or "yes but only when authenticated"?
