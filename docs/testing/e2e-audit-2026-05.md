**Status:** DRAFT — pending Tim review of verdicts before any deletes/downgrades are filed as beads issues.
**Scope:** All 48 E2E spec files (33 full + 14 smoke + 1 root), 193 `test()` blocks total.
**Method:** 5 parallel domain-sliced subagents read every spec body, cross-referenced existing unit/integration coverage, and classified per `test()` block by bug class + verdict. ## TL;DR

- **1 confirmed class-J violation** + **1 structural class-J risk** + **1 needs-verification** — Tim's intuition that "tests of external services have been sneaking in" is confirmed.
- **~10 specs are wholly DOWNGRADE / DELETE.** The biggest single win is `public-routes-audit.spec.ts` (20 tests) where 17 blocks duplicate existing `middleware.test.ts` unit coverage almost verbatim — entire spec collapses to ~3 line additions to one `it.each` array.

- **CI cost reductions expected:**

  - `public-routes-audit.spec.ts` = 20 browser starts per CI run for what is integration-testable.
  - `comment-actions.spec.ts` = 10 sequential `loginAs` cycles per CI run, all class-B/E.

## Class-J Violations Inventory (Tim's flagged concern)

### CONFIRMED: `e2e/full/discord-dm-preferences.spec.ts` — "Send test DM" test (lines 160–192)

- **What it does:** Clicks "Send test DM" → `testDiscordDmAction` server action → `sendDm()` in `src/lib/discord/client.ts` → unconditional `fetch("https://discord.com/api/v10/users/@me/channels")` against Discord's production API with the e2e-fake bot token. The test explicitly relies on Discord returning a non-200 to verify our error-rendering path.
- **Rule #17 diagnostic:** "If this ran against production with real credentials, would the same code pass?" → No (different behavior). Confirmed violation.
- **Replacement target EXISTS:** `src/app/(app)/settings/connected-accounts/test-discord-dm-action.test.ts` already covers all branches (`not_authenticated`, `not_linked`, `not_configured`, `ok`, `blocked`, `rate_limited`, `transient`) with `sendDm` mocked at the boundary.
- **Action:** DELETE the test block; close PP-iak as superseded by audit.

### STRUCTURAL RISK: `e2e/full/admin-discord-integration.spec.ts` — Validate buttons (latent)

- **What it does:** Spec asserts that two "Validate" buttons render in the form, but does NOT click them. The buttons wire to `probeBotToken()` / `probeServerMembership()` which fire live `fetch("https://discord.com/api/v10/users/@me")` and `fetch("https://discord.com/api/v10/guilds/{id}")` with no mock.
- **Why it matters:** One future `await button.click()` test added to this file = instant Class-J violation with no warning. The spec is one PR away from joining the violation list.
- **Action:** DOWNGRADE-integration the entire spec (smoke version already covers the page-render). Move "switch disabled when no token" assertion to an integration test against `actions.ts` with `probeBotToken` mocked at the fetch boundary.

### NEEDS VERIFICATION: `e2e/smoke/image-upload.spec.ts`

- **What it does:** Uploads an image via `ImageUploadButton` → exercises Supabase Storage + Vercel Blob.
- **Verification question:** Does `pnpm run smoke` against the local stack use Supabase local storage (owned) or proxy to live Vercel Blob? If local-only, NO class-J. If live Blob, YES class-J.
- **Action:** Synthesis must check `src/lib/storage/*` and `next.config.ts` for image-upload routing before final verdict. Test 1 ("upload image and persist") is independently DELETE-inert (the test's own code admits "to verify persistence we need to login first" and then doesn't). Test 2 ("authenticated user sees uploaded image") is class-F if local-stack-only.

## Phase 4 — Rule amendment proposals (DRAFT)

Audit confirms rule #11 is the structural driver of bloat. Proposed replacement, gated on Tim review:

### Proposed Rule #11 (replaces existing)

> **Interaction Coverage**: Every clickable user-facing element must be exercised by at least one test at the cheapest layer that catches its bug class. Multi-step user journeys (login → mutate → verify across pages) require E2E; Server Action wiring, permission checks, and DB query correctness require integration tests with PGlite; pure form-state and UI logic require RTL unit tests. Smoke E2E covers "page renders without 500" and layout regression.

### Proposed Rule #17 tightening (append to existing)

> External services other than our owned local stack (Mailpit, PGlite, local Supabase including local Storage, local Vercel Blob proxy if configured) MUST be mocked at the SDK boundary. Tests that drive live Discord webhooks, real OAuth provider redirects, vendor email-template parsing, or any production third-party API endpoint are class-J violations — delete and replace with boundary-mocked unit/integration coverage. Diagnostic: grep your E2E spec for `https://` outside of localhost and our owned-domain URLs.
