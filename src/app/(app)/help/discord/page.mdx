## What it does

Discord powers two features in PinPoint:

- **Sign in with Discord** — once enabled by an admin, members can authenticate with their Discord account instead of (or in addition to) an email and password.
- **Direct-message notifications** — members who connect their Discord account can receive issue updates as DMs from the PinPoint bot, alongside (or instead of) email and in-app notifications.

Both features depend on the same admin-side configuration. They can be set up independently per user from **Settings**.

## How users connect their account

1. Go to **Settings → Connected Accounts → Connect Discord** and authorize PinPoint.
2. Use **Send test DM** under the Discord row to verify the bot can reach your inbox.
3. Open **Settings → Notifications** and pick which events you want as DMs. Discord starts with the same per-event defaults as email — assigned issues, mentions, and new issues are on; status changes and comments are off. Adjust per-event from there.

You can disconnect Discord at any time from the same Connected Accounts row — you'll be asked to confirm. Disconnecting stops DMs immediately; re-link to resume.

## Setting up the Discord bot (first-time admin setup)

Before the admin form has anything to accept, you need a Discord bot that lives in your server. This is a one-time job in the [Discord developer portal](https://discord.com/developers/applications).

1. **Create the application and bot.** In the developer portal, create a new application, open the **Bot** tab, and copy the **bot token** — this is the value you'll paste into PinPoint's **Bot token** field. Treat it like a password; anyone with it can act as your bot.
2. **Add the bot to your server.** Open **OAuth2 → URL Generator**, tick the **`bot`** scope, and leave the permission checkboxes unset — PinPoint only sends direct messages, which needs no server, channel, or privileged-intent permissions beyond the bot simply being a member. Open the generated URL, pick your server, and authorize. The bot can DM any member who shares the server with it and hasn't blocked DMs from server members.
3. **Get the Server (Guild) ID.** In Discord, enable **User Settings → Advanced → Developer Mode**, then right-click your server icon and choose **Copy Server ID**. That value goes in the admin form's **Server ID** field, and validation confirms the bot is a member of it.
4. **Create a non-expiring invite link (optional).** Right-click a channel or the server → **Invite People → Edit invite link**, set **Expire After** to **Never** and **Max Number of Uses** to **No limit**, then copy it. Paste it into the admin form's **Invite link** field; PinPoint surfaces it to users when a DM fails because they aren't in the server.

## How admins enable it

The admin page is a single form with three fields and inline validation.

1. Go to **Integrations → Discord** (the Integrations link is in the user dropdown, admins only).
2. Paste the bot token from the Discord developer portal into **Bot token** and click **Validate** next to it. A green check appears when Discord accepts the token.
3. Enter the **Server ID** for your Discord server and click **Validate** next to it — this confirms the bot is a member of that server. Optionally fill **Invite link**; it's surfaced to users when DMs fail because they aren't in the server.
4. Flip the **Integration enabled** switch. The switch stays disabled until both the token and server ID validate cleanly.
5. Click **Save changes**. Tokens are written to Supabase Vault — you'll never see the saved value again, only re-paste a fresh one to rotate.

Without a saved token the "Sign in with Discord" button stays hidden for everyone, and Discord is omitted from each user's notification preferences.

## Local development and worktrees (env-var seed path)

Pasting a fresh token into the admin form for every new worktree gets tedious. For local development you can seed the bot config from environment variables instead, so a new worktree picks up credentials automatically.

- Set any of these in the worktree's `.env.local`:
  - `DISCORD_BOT_TOKEN` — the bot token (required to seed anything)
  - `DISCORD_GUILD_ID` — the Server ID
  - `DISCORD_INVITE_LINK` — the non-expiring invite link
- The seed step (`pnpm run db:_seed-discord`, run as part of the DB seed pipeline) only acts when **no bot token is saved yet** — the whole step is skipped once the row has a token. On that first seed it creates a Vault secret for the token and, in the same write, fills `guild_id` / `invite_link` from env only where those columns are still null. It never overwrites a value an admin already saved. (Corollary: once a token exists, env vars won't backfill the Server ID or invite link later even if those columns are blank — set them in the admin form.)
- **The database is the source of truth after that first seed.** Env vars are a bootstrap convenience, not a runtime override — once the row has a token, the app reads the DB and ignores these env vars. To change a seeded value later, edit it in the admin form (or clear the row and re-seed).

Sign in with Discord (the OAuth login button, separate from the bot) is driven by `DISCORD_CLIENT_ID` / `DISCORD_CLIENT_SECRET` in `.env.local`, wired through `supabase/config.toml`.

> **Gotcha: restart Supabase after editing `.env.local`.** The `env(...)` substitution in `config.toml` — including the Discord OAuth `client_id` / `secret` — is resolved only when the Supabase containers **start**. Editing `.env.local` while Supabase is already running has no effect until you restart it (`supabase stop && supabase start` in the worktree). If the "Sign in with Discord" button stays hidden after you add the client credentials, an un-restarted stack is the usual cause.

## Troubleshooting

- **"Sign in with Discord" button is missing on the login page.** An admin hasn't enabled the integration yet. Ask an admin to set the bot token and flip the Enabled switch.
- **DMs aren't arriving.** The bot can only DM users who share a server with it. Make sure you're in the configured Discord server — admins can share the invite link from the Integrations page. If you've blocked DMs from server members in your Discord privacy settings, allow them.
- **"Send test DM" returns "blocked".** Same root cause as above — usually privacy settings or not sharing a server with the bot.
- **Bot health shows anything other than "healthy".** Possible values are `unknown`, `healthy`, and `degraded`. Re-check the bot token under Integrations and make sure the bot user hasn't been removed from the server.
